## Описание решения

Система решает задачу **автоматического вызова транспорта на склады** на основе прогноза объёмов отгрузок. Продукт включает три взаимосвязанных компонента:

| Компонент | Что делает |
|-----------|-----------|
| **ML-модель** | Прогнозирует объём отгрузок по каждому маршруту на 6 часов вперед |
| **MILP-оптимизатор** | На основе прогнозов формирует минимально-затратный план диспетчеризации: какие машины, куда и когда отправить |
| **Python-бэкенд** | Предоставляет REST API для дашборда |
| **Веб-сервис** | Предоставляет дашборд для операторов склада |

**Логика работы:**

```
Исторические данные (30-мин слоты)
         │
         ▼
  LightGBM-модели  ──►  Прогноз отгрузок на 6 ч
         │
         ▼
  MILP-оптимизатор  ──►  План: сколько машин × тип × горизонт
         │
         ▼
  POST /call  ──►  Заявка на вызов транспорта (JSON)
```

## ML-модель прогноза

### Задача

Для каждого маршрута (`route_id`) и момента времени `t` предсказать объёмы отгрузок в окнах:

| Горизонт | Шаги модели | Смещение |
|----------|-------------|----------|
| -1.5 ч–1 ч | step_1      | +30 мин  |
| -1 ч–1 ч | step_2      | +60 мин |
| и т.д.    |  и т.д.    | и т.д. |
| 4–6 ч    | step_12     | +270 мин |

### Feature Engineering

Более **100 признаков**:

- **Временны́е**: час, день недели, `hour_sin/cos`, `dow_sin/cos`, `halfhour_slot`
- **Лаги таргета**: 1, 2, 4, 8, 16, 48, 96, 192, 336 шагов назад
- **Скользящие статистики**: mean и std окнами 4, 8, 16, 48, 96, 336
- **EMA**: экспоненциальное скользящее среднее span 8 и 24
- **Межмаршрутные**: офисный агрегат (office_lag_1, office_roll_mean_8)
- **Взаимодействия**: произведение лага на часовой sin

### Модели

Используется ансамбль **LightGBM step-моделей**. Каждый step — отдельная модель регрессии, обученная предсказывать `target_step_i` шаг вперёд. Модели хранятся в `backend/models/` в формате `joblib`.

### Горизонт и периодичность

- **Горизонт уверенного прогноза**: **6 ч** (уверенность обеспечивается за счет доверительного интервала)
- **Шаг прогноза**: **пересекающиеся окна** -1,5ч -- 0.5ч, -1ч -- 1ч, и т.д. до 4ч -- 6ч
- **Периодичность обновления**: каждые **30 минут** (один временной слот)
- При каждом запросе система автоматически подтягивает последние доступные данные

## Методика расчёта ДИ

### Split Conformal Prediction

В задаче прогнозирования спроса на маршрутах транспортировки важно не только получить точечный прогноз, но и оценить диапазон возможных значений (интервал неопределённости), чтобы минимизировать риски недогруза/перегруза и повысить надёжность автоматических решений.

Для построения доверительных интервалов используется метод **split conformal prediction**, гарантирующий строгое покрытие: фактический спрос попадает в интервал не менее чем в $\alpha$% случаев.

#### Калибровочные остатки

* Для каждого маршрута и горизонта собирается калибровочная выборка: абсолютные ошибки модели на отложенных данных.

* В нашем проекте используется **нормализованный скор**:
  
  $$s_i = \frac{\vert y_i - \hat{y}_i \vert}{\hat{y}_i + 1}$$
  
  где $y_i$ — фактический спрос, $\hat{y}_i$ — прогноз.

* Для каждого (route_id, horizon) хранится массив таких скоров (обычно 200+ на пару).

#### Построение интервала

1. Для заданного уровня доверия $\alpha$ (например, 0.9) вычисляется квантиль $q_\alpha$ по калибровочным скором:
   * Используется формула ceiling-index: $q_\alpha = \text{quantile}(\text{scores}, \lceil (n+1)\alpha \rceil / n, \text{method='higher'})$
   * Перед вычислением квантиля применяется **winsorizing**: отбрасываются верхние N% скоров, чтобы исключить аномалии.

2. Полученный квантиль $q_\alpha$ денормализуется под текущий прогноз:
   
   $$\text{margin} = q_\alpha \cdot (\hat{y} + 1)$$

3. Итоговый доверительный интервал:
   
   $$[\hat{y} - \text{margin}, \hat{y} + \text{margin}]$$

4. Дополнительно, для операционной выполнимости, $\text{margin}$ можно ограничить сверху: $\text{margin} \leq 0.5 \cdot \hat{y}$ (не более заданного % прогноза).

### Пример расчёта

Пусть прогноз на 2 часа для маршрута 190: $\hat{y} = 135$.

* Квантиль $q_{0.9}$ по калибровочным скором: $1.35$
* $\text{margin} = 1.35 \times (135 + 1) = 184$ (до winsor/cap)
* После winsorizing и кэпа: $\text{margin} = \min(68, 0.5 \times 135) = 68$
* Итоговый ДИ: $[67, 203]$

### Почему так?

* **Нормализация** защищает от масштабных выбросов на малых маршрутах.
* **Winsorizing** убирает аномалии данных.
* **Кэп** используется, чтобы система не выдавала неадекватно широкий интервал.

## Оптимизатор диспетчеризации

### Постановка задачи (MILP)

Файл `backend/optimizer_horizons.py` формулирует задачу минимизации затрат:

$$\min \sum_{r,v,t} \text{Cost}_{v,r} \cdot x_{r,v,t} + \sum_{r,t} P_{\text{empty}} \cdot u_{r,t} + \sum_{r,t} P_{\text{wait}} \cdot s_{r,t}$$

**Ограничения:**
1. **Баланс потока**: `y[r,t] + s[r,t] − s[r,t−1] = D[r,t]`
2. **Вместимость ТС**: `Σ Cap_v · x[r,v,t] = y[r,t] + u[r,t]`
3. **Парк**: `Σ_r x[r,v,t] ≤ MaxFleet[v,t]`
4. `x ∈ Z≥0`, `y, s, u ≥ 0`

### Горизонты планирования

Базово используется разбивка горизонтов по **2 часа**

| Метка | Смещение | Описание |
|-------|----------|----------|
| A: now | 0 мин | Текущий момент — немедленная отправка |
| B: +2h | +120 мин | Следующий слот |
| C: +4h | +240 мин | Через четыре часа |
| D: +6h | +360 мин | Дальний горизонт |

При необходимости можно изменить гранулярность на **1 ч** либо **0.5 ч** в настройках бизнес-правил. Для этого к прогнозам ML-моделей применяется **математическая деконволюция**, разбивающая прогнозы на более мелкие интервалы.

| `POST` | `/dispatch` | Полный план для всех маршрутов склада |
| `POST` | `/optimize` | Прогноз + план для одного маршрута |

#### Пример: POST /dispatch

```json
{
  "warehouse_id": "w03",
  "timestamp": "2024-12-01T10:00:00"
}
```

Ответ содержит массив `routes` — по одному объекту на маршрут с вложенным `plan[]`.

#### Пример: POST /call

```json
{
  "warehouse_id": "w03",
  "route_id": "202",
  "timestamp": "2024-12-01T10:00:00"
}
```

Возвращает готовую заявку на вызов транспорта с типами ТС, количеством, адресом и временем.

## Бизнес-допущения

В рамках прототипа приняты следующие упрощения:

| # | Допущение | Обоснование |
|---|-----------|-------------|
| 1 | **Вместимость** — единица измерения условная (не кг/м³). Каждый тип ТС имеет фиксированную вместимость в «единицах товара». | Данные конкурса не содержат физических объёмов. |
| 2 | **Одна стоимость** на тип ТС. Нет учёта пробок, топлива, толерантности водителя. | Упрощение для бизнес-логики; в проде заменяется реальными тарифами. |
| 3 | **Расстояния по маршрутам фиксированы**. | В прототипе нет real-time маршрутизации. |
| 4 | **Парк транспорта ограничен**. MILP не допускает превышение. | Имитирует реальный транспортный парк складов. |
| 5 | **Один цикл прогноза = 30 минут**. | Соответствует гранулярности тренировочных данных. |
| 6 | **Все склады** не конкурируют за парк (каждый склад — независимый офис). | Межскладная конкуренция за ТС в прототипе не реализована. |
| 7 | **Остатки на складе** берутся из базы данных приложения, не из real-time ERP. | В проде — интеграция с WMS. |
| 8 | **Модель обучалась** на данных командного трека хакатона (`train_team_track.parquet`). | Производительность на реальных данных требует дополнительной валидации. |
| 9 | **ТС отправляются раз в заданный интервал** | В проде можно добавить неравномерные отправки для более оптимальных решений. |

## Метрики качества

### ML-метрики (соревновательные)

| Метрика | Описание |
|---------|----------|
| **WAPE** (Weighted Absolute Percentage Error) | $\frac{\sum \vert y - \hat{y} \vert}{\sum y}$ — основная метрика конкурса |
| **RBias** | Относительное смещение предсказания, штрафует систематические ошибки |

Целевая функция: минимизация `WAPE + λ·|RBias|`.

### Бизнес-метрики системы

| Метрика | Описание | Как считать                                           |
|---------|----------|-------------------------------------------------------|
| **Общие затраты** | Цена отправки + штрафы | `цена_отправки + штраф_простоя + штраф_недогруз`      |
| **Цена одной доставки** | Цена отправки одной единицы товара | `общие_затраты / отправлено_товара`                   |
| **Вероятность достаточности транспорта** | Позволяет оценить, насколько вероятно отправки будут покрыты доступным и прибывающим парком | По плану оптимизатора                                 |
| **Коэффициент утилизации парка** | Отношение требуемой вместимости к общей вместимости **доступных** ТС. Отражает, какая доля транспорта простаивает или сколько не хватает  | `итого_товаров / Σ(вместимость_ТС)`                   |
| **Нехватка вместимости** | Разница требуемой вместимости и общей вместимости **доступных** ТС. Отражает, сколько единиц транспорта не хватает | `итого_товаров - Σ(вместимость_ТС)`                   |
| **Коэффициент загрузки ТС** | Отношение отправленных единиц к общей вместимости **отправленных** ТС. Позволяет оценить недогруженность транспорта | `отправлено_товаров / Σ(вместимость_отправленных_ТС)` |

### Бейзлайны для сравнения

- **Наивные оптимизаторы**: распределяют транспорт без смешивания типов ТС: первый полностью покрывает спрос, второй экономит рейсы и допускает недопокрытие.
- **Наивное прогнозирование**: считать, что прогноз всегда сбудется и доверительный интервал считать не нужно

## Пути развития продукта

### Улучшение прогноза

| Направление | Описание |
|-------------|----------|
| **Внешние данные** | Данные о праздниках, акциях, промо-кампаниях WB — ключевые факторы пиков спроса |
| **Погодные данные** | Влияют на скорость доставки и готовность водителей |
| **Данные из ERP/WMS** | Real-time остатки товара, статус сборки заказов |
| **Трафик и длина маршрута** | Яндекс Карты API для корректировки времени в пути |
| **Неравномерная отправка транспорта** | В проде можно добавить неравномерные отправки для более оптимальных решений |

### Масштабирование системы

| Направление | Описание |
|-------------|----------|
| **Онлайн-обучение** | Дообучение модели в режиме реального времени при появлении новых данных |
| **Мультисклад MILP** | Совместная оптимизация парка между несколькими складами конкурирующими за машины |
| **Re-routing** | Перенаправление уже выехавших ТС при изменении прогноза |
| **Интеграция с ТМС** | Автоматическая отправка заявок в транспортно-менеджерскую систему |
| **Push-уведомления** | Алерты диспетчерам при критическом изменении прогноза |
| **A/B тестирование** | Параллельный запуск системы и ручного управления для сравнения метрик |

### Операционная зрелость

- **Мониторинг**: Prometheus + Grafana для отслеживания WAPE в production
- **Data drift detection**: автоматическое обнаружение смещения распределения входных данных
- **Explainability**: SHAP-значения для объяснения прогнозов диспетчерам
- **CI/CD**: автоматический ре-тренинг и валидация моделей при обновлении данных
