## Описание решения

Система решает задачу **автоматического вызова транспорта на склады** на основе прогноза объёмов отгрузок. Продукт включает три взаимосвязанных компонента:

| Компонент | Что делает |
|-----------|-----------|
| **ML-модель** | Прогнозирует объём отгрузок по каждому маршруту на три горизонта вперёд: 0–2 ч, 2–4 ч, 4–6 ч |
| **MILP-оптимизатор** | На основе прогнозов формирует минимально-затратный план диспетчеризации: какие машины, куда и когда отправить |
| **Веб-сервис** | Предоставляет REST API и дашборд для операторов склада |

**Логика работы:**

```
Исторические данные (30-мин слоты)
         │
         ▼
  LightGBM-модели  ──►  Прогноз отгрузок на 6 ч
         │
         ▼
  MILP-оптимизатор  ──►  План: сколько машин × тип × горизонт
         │
         ▼
  POST /call  ──►  Заявка на вызов транспорта (JSON)
```

## ML-модель прогноза

### Задача

Для каждого маршрута (`route_id`) и момента времени `t` предсказать объёмы отгрузок в трёх окнах:

| Горизонт | Шаги модели | Смещение |
|----------|-------------|----------|
| 0–2 ч    | step_4      | +30 мин  |
| 2–4 ч    | step_8      | +150 мин |
| 4–6 ч    | step_12     | +270 мин |

### Feature Engineering

Файл `backend/ml_prediction.py` реализует более **100 признаков**:

- **Временны́е**: час, день недели, `hour_sin/cos`, `dow_sin/cos`, `halfhour_slot`
- **Лаги таргета**: 1, 2, 4, 8, 16, 48, 96, 192, 336 шагов назад
- **Скользящие статистики**: mean и std окнами 4, 8, 16, 48, 96, 336
- **EMA**: экспоненциальное скользящее среднее span 8 и 24
- **Межмаршрутные**: офисный агрегат (office_lag_1, office_roll_mean_8)
- **Взаимодействия**: произведение лага на часовой sin

### Модели

Используется ансамбль **LightGBM step-моделей** (эксперимент `exp_best_features_full`). Каждый step — отдельная модель регрессии, обученная предсказывать `target_step_i` шаг вперёд. Модели хранятся в `backend/models/exp_best_features_full/` в формате `joblib`.

### Горизонт и периодичность

- **Горизонт уверенного прогноза**: **0–2 ч**
- **Рабочий горизонт планирования**: **6 ч**
- **Периодичность обновления**: рекомендуется каждые **30 минут** (один временной слот)
- При каждом запросе система автоматически подтягивает последние доступные данные (`predict_lazy`)

## Методика расчёта доверительного интервала для прогноза спроса

### Проблема

В задаче прогнозирования спроса на маршрутах транспортировки важно не только получить точечный прогноз, но и оценить диапазон возможных значений (интервал неопределённости), чтобы минимизировать риски недогруза/перегруза и повысить надёжность автоматических решений.

### Используемый подход: Split Conformal Prediction

Для построения доверительных интервалов используется метод split conformal prediction, гарантирующий строгое покрытие: фактический спрос попадает в интервал не менее чем в α% случаев.

#### Калибровочные остатки
- Для каждого маршрута и горизонта собирается калибровочная выборка: абсолютные ошибки модели на отложенных данных.
- В нашем проекте используется **нормализованный скор**:
  $$ s_i = \frac{|y_i - \hat{y}_i|}{\hat{y}_i + 1} $$
  где $y_i$ — фактический спрос, $\hat{y}_i$ — прогноз.
- Для каждого (route_id, horizon) хранится массив таких скоров (обычно 200+ на пару).

#### Построение интервала
1. Для заданного уровня доверия $\alpha$ (например, 0.9) вычисляется квантиль $q_\alpha$ по калибровочным скором:
   - Используется формула ceiling-index: $q_\alpha = \text{quantile}(scores, \lceil (n+1)\alpha \rceil / n, \text{method='higher'})$
   - Перед вычислением квантиля применяется **winsorizing**: отбрасываются верхние 5% скоров (95-й перцентиль), чтобы исключить аномалии.
2. Полученный квантиль $q_\alpha$ денормализуется под текущий прогноз:
   $$ \text{margin} = q_\alpha \cdot (\hat{y} + 1) $$
3. Итоговый доверительный интервал:
   $$ [\hat{y} - \text{margin},\ \hat{y} + \text{margin}] $$
4. Дополнительно, для операционной выполнимости, margin ограничивается сверху: $$ \text{margin} \leq 0.5 \cdot \hat{y} $$ (не более 50% прогноза).

#### Fallback-логика
- Если для пары (route_id, horizon) мало данных, используется агрегат по горизонту, затем глобальный пул.

### Пример расчёта

Пусть прогноз на 2 часа для маршрута 190: $\hat{y} = 135$.
- Квантиль $q_{0.9}$ по калибровочным скором: 1.35
- margin = $1.35 \times (135 + 1) = 184$ (до winsor/cap)
- После winsorizing и кэпа: margin = min(68, 0.5*135) = 68
- Итоговый ДИ: $[67, 203]$

### Почему так?
- **Нормализация** защищает от масштабных выбросов на малых маршрутах
- **Winsorizing** убирает аномалии данных
- **Кэп** гарантирует, что система не выдаст неадекватно широкий интервал

## Оптимизатор диспетчеризации

### Постановка задачи (MILP)

Файл `backend/optimizer_horizons.py` формулирует задачу минимизации затрат:

$$\min \sum_{r,v,t} \text{Cost}_{v,r} \cdot x_{r,v,t} + \sum_{r,t} P_{\text{empty}} \cdot u_{r,t} + \sum_{r,t} P_{\text{wait}} \cdot s_{r,t}$$

**Ограничения:**
1. **Баланс потока**: `y[r,t] + s[r,t] − s[r,t−1] = D[r,t]`
2. **Вместимость ТС**: `Σ Cap_v · x[r,v,t] = y[r,t] + u[r,t]`
3. **Парк**: `Σ_r x[r,v,t] ≤ MaxFleet[v,t]`
4. `x ∈ Z≥0`, `y, s, u ≥ 0`

### Горизонты планирования

| Метка | Смещение | Описание |
|-------|----------|----------|
| A: now | 0 мин | Текущий момент — немедленная отправка |
| B: +2h | +120 мин | Следующий слот |
| C: +4h | +240 мин | Через четыре часа |
| D: +6h | +360 мин | Дальний горизонт |

### Типы транспортных средств

| Тип | Вместимость (ед.) | Стоимость (руб/км) | Фикс. затраты |
|-----|--------------------|---------------------|---------------|
| Lada Largus | 3 | 5.65 | 500 |
| ГАЗель NN ЦМФ | 6 | 8.52 | 800 |
| AMBERTRUCK JL | 10 | 7.74 | 1 200 |
| Валдай 8 | 14 | 8.90 | 1 500 |
| Hyundai Mighty EX8 XL | 18 | 10.84 | 2 000 |
| ISUZU ELF 7.5 Long | 20 | 12.77 | 2 500 |

### Выходной план

Результат — таблица строк, одна строка на пару (горизонт × тип ТС):

```
horizon | vehicle_type | vehicles_count | demand | actually_shipped | cost_total
A: now  | ГАЗель NN    |       3        |  17.0  |       18.0       |  4 800
```

## API-сервис

### Запуск

```bash
cd backend
python -m venv .venv
.venv\Scripts\activate          # Windows
pip install -r requirements.txt
uvicorn main:app --port 8000 --reload
```

Swagger UI: [http://localhost:8000/docs](http://localhost:8000/docs)

### Основные эндпоинты

| Метод | URL | Описание |
|-------|-----|----------|
| `POST` | `/dispatch` | Полный план для всех маршрутов склада |
| `POST` | `/optimize` | Прогноз + план для одного маршрута |
| `POST` | `/call` | Сформировать заявку на вызов ТС (горизонт A) |
| `GET` | `/health` | Статус сервиса и загруженных моделей |
| `GET/POST` | `/vehicles` | Просмотр и редактирование парка |
| `GET/POST` | `/warehouses` | Список складов |
| `PATCH` | `/settings` | Изменить штрафы и параметры |

#### Пример: POST /dispatch

```json
{
  "warehouse_id": "w03",
  "timestamp": "2024-12-01T10:00:00"
}
```

Ответ содержит массив `routes` — по одному объекту на маршрут с вложенным `plan[]`.

#### Пример: POST /call

```json
{
  "warehouse_id": "w03",
  "route_id": "202",
  "timestamp": "2024-12-01T10:00:00"
}
```

Возвращает готовую заявку на вызов транспорта с типами ТС, количеством, адресом и временем.

## Бизнес-допущения

В рамках прототипа приняты следующие упрощения:

| # | Допущение | Обоснование |
|---|-----------|-------------|
| 1 | **Вместимость** — единица измерения условная (не кг/м³). Каждый тип ТС имеет фиксированную вместимость в «единицах товара». | Данные конкурса не содержат физических объёмов. |
| 2 | **Одна стоимость за км** на тип ТС. Нет учёта пробок, топлива, толерантности водителя. | Упрощение для бизнес-логики; в проде заменяется реальными тарифами. |
| 3 | **Расстояния фиксированы** в `route_distances.json`. | В прототипе нет real-time маршрутизации. |
| 4 | **Парк ограничен** количеством в `vehicles.json`. MILP не допускает превышение. | Отражает реальный физический парк склада. |
| 5 | **Один цикл прогноза = 30 минут**. Данные обновляются с каждым новым слотом. | Соответствует гранулярности тренировочных данных. |
| 6 | **Горизонт уверенного прогноза** — 2 ч (шаг step_1). Прогнозы на 4–6 ч используются для резервирования парка, не для немедленного вызова. | Точность моделей падает с ростом горизонта. |
| 7 | **Все склады в одном офисном кластере** не конкурируют за парк (каждый склад — независимый офис). | Межскладная конкуренция за ТС в прототипе не реализована. |
| 8 | **Остатки на складе** (`ready_to_ship`) берутся из `route_distances.json`, не из real-time ERP. | В проде — интеграция с WMS. |
| 9 | **Модель обучалась** на данных командного трека хакатона (`train_team_track.parquet`). | Производительность на реальных данных требует дополнительной валидации. |
| 10 | **Заявка вызова** формируется только по горизонту A (текущий момент). Горизонты B–D — предварительное резервирование. | Операционная логика: немедленный вызов только при наличии спроса сейчас. |

## Метрики качества

### ML-метрики (соревновательные)

| Метрика | Описание |
|---------|----------|
| **WAPE** (Weighted Absolute Percentage Error) | `Σ|y − ŷ| / Σy` — основная метрика конкурса |
| **RBias** | Относительное смещение предсказания, штрафует систематические ошибки |

Целевая функция: минимизация `WAPE + λ·|RBias|`.

### Бизнес-метрики системы

| Метрика | Описание | Как считать |
|---------|----------|-------------|
| **Fill Rate** | Доля спроса, покрытая отправками | `actually_shipped / demand_total` |
| **Недогруз ТС** | Средняя доля пустого пространства в кузовах | `empty_capacity_units / total_capacity` |
| **Стоимость на единицу товара** | `cost_total / actually_shipped` | По плану оптимизатора |
| **Опоздание отгрузки** | Доля заявок, переходящих в следующий горизонт (leftover) | `leftover_stock > 0` |
| **Утилизация парка** | `vehicles_dispatched / vehicles_available` | По типу ТС |

### Бейзлайн для сравнения

- **Наивный прогноз**: значение за аналогичный период прошлой недели (`lag_336`)
- **Правило большого пальца**: всегда отправлять X ТС одного типа без учёта прогноза
- **Текущая ручная диспетчеризация** (если есть исторические данные по фактическим вызовам)

### Тестирование

```bash
cd backend
pytest tests/ -v
```

- `test_ml_prediction.py` — инференс модели на синтетических данных
- `test_optimizer_horizons.py` — корректность MILP-решения, баланс потока

| **Онлайн-обучение** | Дообучение модели в режиме реального времени при появлении новых данных |
| **Мультисклад MILP** | Совместная оптимизация парка между несколькими складами конкурирующими за машины |
| **Re-routing** | Перенаправление уже выехавших ТС при изменении прогноза |
| **Confidence-интервалы** | Стохастическая оптимизация с учётом неопределённости прогноза |
| **Интеграция с ТМС** | Автоматическая отправка заявок в транспортно-менеджерскую систему |
| **Push-уведомления** | Алерты диспетчерам при критическом изменении прогноза |
| **A/B тестирование** | Параллельный запуск системы и ручного управления для сравнения метрик |

### Операционная зрелость

- **Мониторинг**: Prometheus + Grafana для отслеживания WAPE в production
- **Data drift detection**: автоматическое обнаружение смещения распределения входных данных
- **Explainability**: SHAP-значения для объяснения прогнозов диспетчерам
- **CI/CD**: автоматический ре-тренинг и валидация моделей при обновлении данных
